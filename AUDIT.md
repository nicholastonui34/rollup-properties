# Rollup Properties — Codebase Audit

**Scope:** Phase 0 (orientation) + Phase 1 (spec audit). Read-only — no application code was changed.
**Date:** 2026-07-18

---

## 1. Orientation Summary

### Stack
Next.js 16.2.10 (App Router, React 19.2.4, TypeScript strict), Tailwind CSS v4 + shadcn/ui (`components.json`, `radix-ui` primitives), Prisma 6.19 against Neon Postgres, Auth.js v5 beta (`next-auth@5.0.0-beta.31`) with a single `Credentials` provider (phone-or-email + bcryptjs), Leaflet 1.9 + `react-leaflet` 5 for maps, Zod 4 for all input validation. `dev`/`build` scripts explicitly force `--webpack` (OneDrive/Turbopack gotcha, matches memory). Cloudinary and Paystack are integrated as hand-rolled `fetch`-based clients in `src/lib/cloudinary.ts` and `src/lib/paystack.ts` — no SDK dependency for either, which keeps the bundle lean and secrets server-side only. There is no `src/middleware.ts`; all route protection is done per-route via `auth()` checks in layouts (`src/app/dashboard/layout.tsx`, `src/app/(admin)/admin/layout.tsx`) and per-mutation guard helpers (`src/lib/auth-guards.ts`: `requireVerifier`, `requireAdmin`) plus inline ownership checks in Server Actions. Deployment: `vercel.json` only declares one cron (`/api/cron/saved-search-alerts`, daily 06:00) — no other Vercel-specific config (no explicit regions/edge runtime declarations).

### Data model (`prisma/schema.prisma`)
`User` (role enum SEEKER/LISTER/VERIFIER/ADMIN, phone-unique, optional email, `idNumber`/`idVerifiedAt` for KYC — no ban/suspend field), `Area` (county/town/name seed table), `Listing` (purpose RENT/SALE, propertyType enum of 8 types, price/deposit/serviceCharge in KES, county/town/area/estate/streetAddress/lat/lng, bedrooms/bathrooms/furnished/amenities string[], 9-state `ListingStatus` enum, verifiedAt/verifiedById/expiresAt/viewCount), `ListingImage` (photos only — no video/tour field), `Verification` (append-only audit log: status/notes/evidenceUrls per review action), `Unlock` (unique per user+listing, linked 1:1 to a `Payment`), `Payment` (provider PAYSTACK/MPESA, status incl. REFUNDED, purpose enum UNLOCK/BUNDLE/FEATURED/SUBSCRIPTION — the latter two are unused placeholders, no supporting models), `SavedListing`, `SavedSearch` (JSON filters + alert flag), `Report`. No `Subscription`/`Plan`/`featured` boolean model exists despite the `FEATURED`/`SUBSCRIPTION` enum values.

### Routes (`src/app/**`, one line each)
- `/` — marketing homepage: hero search bar, trust strip, "how it works", popular Nairobi areas, lister CTA.
- `/login`, `/signup` — Auth.js credentials login/signup forms (route group `(auth)`).
- `/search` — filterable/sortable/paginated listing search, grid or Leaflet map view, save-search affordance; `noindex` (canonical SEO lives on area pages).
- `/rent/[area]`, `/sale/[area]` — SEO landing pages per Nairobi area/purpose, indexable.
- `/listings/[slug]` — public listing detail: gallery, verified badge, price/terms, amenities, contact-unlock panel, favorite/report buttons, `RealEstateListing` JSON-LD.
- `/listings/[slug]/opengraph-image.tsx` — per-listing OG image.
- `/favorites` — seeker's saved listings.
- `/saved-searches` — manage saved searches + alert toggle.
- `/unlocks` — seeker's paid-unlock receipts.
- `/dashboard` — lister dashboard: listings list, KYC status/submission.
- `/dashboard/listings/new`, `/dashboard/listings/[id]/edit` — create/edit listing incl. photo upload, submit-for-review.
- `/dashboard/listings/[id]/leads` — inbox of users who unlocked this listing's contact.
- `(admin)/admin` — verifier/admin dashboard shell (nav to sub-pages).
- `/admin/verifications`, `/admin/verifications/[id]` — verification queue + approve/reject/needs-info with evidence.
- `/admin/listers` — lister KYC review/approve.
- `/admin/payments` — payments ledger + manual "mark refunded".
- `/admin/reports` — reported-listing queue, suspend/reinstate.
- `/api/auth/[...nextauth]` — Auth.js handler.
- `/api/cloudinary/sign`, `/api/cloudinary/sign-evidence` — signed upload endpoints (listing photos vs. verification evidence, separate folders).
- `/api/paystack/webhook` — Paystack `charge.success` webhook, HMAC-verified, idempotent.
- `/payments/callback` — Paystack hosted-checkout redirect handler.
- `/api/cron/saved-search-alerts` — daily digest emailing saved-search matches (Resend, no-ops without API key).
- `/privacy`, `/terms`, `/refund-policy` — legal pages.
- `/sitemap.ts`, `/robots.ts`, `/opengraph-image.tsx` — SEO plumbing.
- `/error.tsx`, `/not-found.tsx` — error/404 boundaries.

### Design system
Tailwind v4 with CSS-variable/OKLCH tokens (`src/app/globals.css`): a deep forest-green primary (`oklch(0.38 0.07 165)`), warm off-white background, and a `gold` accent token used sparingly for badges/CTAs against the primary green — a deliberate "trust + Kenyan warmth" palette rather than shadcn defaults. Typography pairs Geist Sans (body) with Fraunces (a serif, via `--font-display`) for headings, giving marketing pages an editorial feel distinct from the dashboard's utilitarian card layout. Generous rounded corners (`rounded-xl`/`2xl`/`3xl`) and soft shadows throughout. The shadcn component set is intentionally minimal — only `button`, `badge`, `card`, `input`, `label`, `select`, `textarea` exist under `src/components/ui`; everything else (map view, listing cards, search filters, image uploader, admin panels) is bespoke composition on top of those primitives, organized by domain folder (`components/listing`, `components/search`, `components/admin`, `components/site`).

## 3. Fix Plan (milestones, ordered P0 → P1 → P2)

### Milestone F1 — Trust & abuse-protection hardening (P0)
Closes rows: **3d/10e** (rate limiting), **10d** (lister ban).
- Add a `bannedAt DateTime?` (or a `status: ACTIVE|BANNED` enum) to `User`; add an admin `banListerAction` in `admin/listers/actions.ts` requiring a reason, that also cascades to suspend/hide that lister's other LIVE listings; have `authorize()` in `src/lib/auth.ts` reject banned users at login.
- Add basic rate limiting to the highest-risk unverified endpoints: `initiateUnlockAction`, `reportListingAction`, the credentials `authorize()` callback, and the Paystack webhook/callback routes. Since there's no Redis/Upstash in the stack yet, either add `@upstash/ratelimit` + a free Upstash Redis instance, or implement a minimal in-memory/DB-backed per-IP+per-user throttle as an interim measure (acceptable for a single-region Vercel deployment, revisit if scaling to multi-region).
- Scope: mostly additive — one migration, one new admin action, one new small rate-limit utility wired into ~4 call sites. No architectural changes.

### Milestone F2 — Lifecycle status correctness (P1)
Closes rows: **6a/6b/6c**.
- Either (a) rename the single `TAKEN` status's *display* conditionally on `listing.purpose` ("Rented" for RENT, "Sold" for SALE) — cheapest fix, no migration — or (b) add real `RENTED`/`SOLD` enum values and a migration mapping existing `TAKEN` rows by purpose. Recommend (a) first given effort/impact, revisit (b) only if reporting/analytics need to distinguish them at the DB level.
- Fix the listing-detail 404 gate (`listings/[slug]/page.tsx:61`) to let `TAKEN`/`EXPIRED` listings render (for anyone, not just owner/admin) with a clear "no longer available" banner instead of blocking with `notFound()`. Keep `notFound()` for DRAFT/SUBMITTED/IN_VERIFICATION/REJECTED/NEEDS_INFO/SUSPENDED (pre-publication or actioned states that genuinely shouldn't be public).
- Scope: small, contained to 2-3 files.

### Milestone F3 — Transparent data completeness (P1)
Closes rows: **4a** (partial).
- Add a `sizeSqm Int?` (or `Float?`) column to `Listing`; surface it in the create/edit form and the detail page stat row alongside bed/bath/furnished.
- Replace conditional-omission rendering (`depositKes != null && (...)`, `serviceChargeKes != null && (...)`) with an always-rendered row that shows the value or literal "Not provided" text, per spec.
- Scope: one migration + edits to 2 form files + 1 detail-page section.

### Milestone F4 — Media hierarchy: tours & video (P1, largest lift)
Closes rows: **1a/1b**, and the "media" portion of **7a**.
- Add `tourEmbedUrl String?` and `videoUrl String?` to `Listing` (simplest: nullable iframe/URL fields, not a full new media-type table, to match the "sketch" style of the rest of the schema).
- Add fields + validation (Zod URL check, plus an allowlist of embeddable domains for the tour iframe as a lightweight XSS guard since it will render in an `<iframe src=...>`) to the listing create/edit form.
- Reorder the listing-detail page media block so a tour/video (if present) renders above the photo carousel, matching the spec's "tour-first hierarchy"; fall back to photo grid when neither exists (current behavior, unchanged).
- Decide/flag hosting strategy: embedding a third-party tour (Matterport/Kuula-style iframe) avoids self-hosting large video files, which is the right call for a Vercel + Cloudinary stack — flag to the user that self-hosted raw video would need Cloudinary's video product (separate from the current image-only integration) or an external video host, and confirm before building either.
- Scope: 1 migration, form changes, detail-page reorder — moderate effort mostly because of the new upload/embed UX, not schema complexity.

### Milestone F5 — Featured placement & subscription scaffolding (P1/P2)
Closes rows: **8b/8c**.
- Add `featuredUntil DateTime?` to `Listing`; add an admin-only action to set/clear it (manual for now, per spec's "do not integrate a new payment rail without confirmation"); boost featured listings in `src/lib/search.ts` `orderBy()` (e.g., `featuredUntil` desc-then-null-last as a primary sort key) and add a "Featured" badge in `listing-card.tsx`.
- Stub a `Subscription`/tier concept only as far as schema (`planTier` on `User` or a small `Plan` lookup table) — do not build billing; explicitly flag the payment-rail decision to the user before any further work here, per spec §8.
- Scope: schema + search ordering is S; any real subscription enforcement (listing-count caps per tier) is a separate, larger future milestone — out of scope until monetization direction is confirmed.

### Milestone F6 — Polish & verification hygiene (P2)
Closes rows: **2a** (badge explanation), **2d** (60-day re-verification flag), **5a** (status filter in search), **5c** (debounce verification), **9a/9b** (flag-only, no code change pending confirmation), plus the two "Eng." rows (lint/build baseline, WCAG pass).
- Add a small info affordance (tooltip or link to an "About verification" section) next to the verified badge.
- Add a scheduled query (reuse the existing cron pattern) that flags `LIVE` listings with `verifiedAt` older than 60 days into an admin view for re-check — doesn't need to auto-unpublish, just surface.
- Expose a lifecycle-status filter/toggle in `/search` (e.g., "show taken/rented" checkbox) to satisfy the "explicit filter" half of row 6c independent of direct-link access.
- Verify (and add if missing) input debounce in `components/search/search-filters.tsx`.
- Before any further build work, run `npm run lint` and `npm run build` once to confirm a clean baseline, and schedule a dedicated accessibility pass.
- Confirm with the user whether non-KES/non-Kenyan-card unlock flows need explicit testing against Paystack's live settings (row 9a) before assuming it "just works."

## Overall read

The codebase is materially more complete and more disciplined than a typical MVP scaffold — server-side authorization is applied consistently and per-mutation (not just at the UI layer), the verification state machine matches the brief closely, the Paystack unlock flow has a correctly-designed idempotent webhook-as-source-of-truth pattern, and the map/search stack is genuinely wired up (Leaflet+OSM, not a stub). The gaps that matter are concentrated exactly where the spec's "trust" framing is strongest: the homepage and refund-policy pages make an explicit, specific promise ("we refund you and ban the lister") that only half exists in code — refunds are ledger-tracked but there is no ban mechanism at all. Combined with the complete absence of rate limiting on public POST endpoints (unlock-initiate, report, login), these are the two P0 items. The tour/video media hierarchy the spec treats as primary is entirely absent from the schema, which is the single largest scoped gap (P1, Milestone F4) but is architecturally straightforward to add given the existing nullable-field, iframe-embed-friendly style of the rest of the model.
